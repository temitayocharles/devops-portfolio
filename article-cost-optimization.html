The Strategy: Find the Waste

Before optimizing, you must identify waste. Most waste falls into three categories:

  • Idle Resources: Running but unused (stopped VMs, disconnected volumes)
  • Oversized Resources: Resource requests 2-3x larger than needed
  • Wrong Purchasing Model: On-demand when commitment-based is cheaper

Let's tackle each systematically.

Strategy 1: Reserved Instances (RI) and Savings Plans

This is the biggest lever. Commitments can save 30-70% on compute costs.

AWS Approach

# Analyze current spend aws ce get-cost-and-usage --time-period Start=2025-01-01,End=2025-12-31 \ --granularity MONTHLY \ --metrics "UnblendedCost" \ --group-by Type=DIMENSION,Key=INSTANCE_TYPE # Calculate RI savings aws ce get-reservation-purchase-recommendation \ --service EC2 \ --lookback-period THIRTY_DAYS \ --term-in-years 1

Strategy

  1. Analyze baseline usage: Which instances run 24/7?
  2. Purchase 1-year RIs for baseline: Save 30-40%
  3. Use Compute Savings Plans for flex: Save 17-25% with flexibility
  4. Run Spot for variable workloads: Save 60-90%
Result: 1-year RI commitment on baseline compute can reduce spend by 35-50% with zero performance impact.

Strategy 2: Kubernetes Right-Sizing

Kubernetes makes it easy to waste resources. Here's how to optimize:

Step 1: Analyze Pod Resource Usage

# Get actual resource usage vs requests kubectl top pods -A --containers # Example output: # NAMESPACE POD CPU(cores) MEMORY(Mi) # default app-1 50m 200Mi # default app-2 10m 50Mi # (But they probably requested 500m/512Mi each!)

Step 2: Right-Size Requests and Limits

Don't guess. Use actual usage metrics:

# BAD: Guessing at resource requirements resources: requests: cpu: 500m # Probably 10x more than needed! memory: 512Mi limits: cpu: 1000m memory: 1Gi # GOOD: Based on actual usage + buffer resources: requests: cpu: 60m # P99 usage: 50m + 20% buffer memory: 256Mi # P99 usage: 200Mi + 28% buffer limits: cpu: 200m # Prevent runaway memory: 512Mi

Step 3: Implement Vertical Pod Autoscaler (VPA)

VPA automatically recommends and applies right-sizing:

# Install VPA helm repo add fairwinds-stable https://charts.fairwinds.com/stable helm install vpa fairwinds-stable/vpa # It then monitors and recommends optimal sizes # Check recommendations: kubectl get vpa -A kubectl describe vpa my-app

Strategy 3: Spot Instances & Preemptible VMs

Use short-term instances for fault-tolerant workloads:

# Kubernetes Spot Configuration apiVersion: apps/v1 kind: Deployment metadata: name: batch-processor spec: template: spec: affinity: nodeAffinity: preferredDuringSchedulingIgnoredDuringExecution: - weight: 100 preference: matchExpressions: - key: karpenter.sh/capacity-type operator: In values: ["spot"] # Prefer Spot tolerations: - key: spot operator: Equal value: "true" effect: NoSchedule containers: - name: app image: my-app:latest # Save: 60-90% vs on-demand

Strategy 4: Storage Optimization

Storage costs add up silently. Optimize by tier and retention:

Tiering Strategy

Storage Tier Use Case Cost Access Pattern
Hot (SSD) Active databases, cache $0.10/GB/mo Frequent access
Warm (HDD) Archives, backups $0.02/GB/mo Infrequent access
Cold (Glacier) Compliance, rarely accessed $0.004/GB/mo Rare access

Implementation

# AWS S3 Lifecycle Policy { "Rules": [ { "Id": "AutoTierData", "Status": "Enabled", "Transitions": [ { "Days": 30, "StorageClass": "STANDARD_IA" }, { "Days": 90, "StorageClass": "GLACIER" } ], "Expiration": { "Days": 730 # Delete after 2 years } } ] } # Result: Reduce storage cost by 60-80%

Strategy 5: Network Optimization

Data transfer costs are often hidden. Optimize:

  • Cross-region data transfer: Minimize by region-localizing workloads
  • Internet egress: Use CDN (CloudFront, Azure CDN) instead of direct egress
  • Unused IPs: Release unattached Elastic IPs (~$3/month each)
  • VPC peering: Cheaper than NAT Gateway for inter-VPC communication

Strategy 6: Database Right-Sizing

Databases often run on oversized instances:

# Check actual database load SELECT sum(heap_blks_read) as heap_read, sum(idx_blks_read) as index_read, sum(heap_blks_hit) / (sum(heap_blks_hit) + sum(heap_blks_read)) as ratio FROM pg_statio_user_tables; # If ratio > 0.99, you're cache-bound and can downsize # If ratio < 0.95, you need more cache
