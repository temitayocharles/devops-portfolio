Observability at Scale: Metrics, Logs, and Traces

Author: Temitayo Charles Published: February 6, 2026 Read Time: 13 minutes

Observability is the difference between operating infrastructure with confidence and operating it blind. As systems grow more complex and distributed, traditional monitoring becomes insufficient. You need observability: the ability to ask arbitrary questions about your system's behavior.

The Three Pillars: Metrics, Logs, Traces

Observability rests on three pillars. Each answers different questions:

Metrics (Time-Series Data)

What to ask: "What's the CPU usage?" "How many requests per second?" "What's the disk space trend?"

Answer speed: Milliseconds. Metrics are sampled, aggregated, and queryable instantly.

Use case: Dashboards, alerting, capacity planning.

Logs (Structured Event Data)

What to ask: "Why did request X fail?" "What happened between timestamp A and B?" "Which service is throwing errors?"

Answer speed: Seconds. Logs are voluminous; querying them takes time.

Use case: Debugging specific incidents, audit trails, security analysis.

Traces (Request Flow)

What to ask: "How long did request X take?" "Which service was slow?" "Where did the request fail?"

Answer speed: Seconds to minutes. Traces require correlation across services.

Use case: Performance debugging, dependency analysis, latency investigation.

Metrics: Prometheus at Scale

Prometheus is the standard for Kubernetes monitoring. Here's how to scale it:

Architecture Pattern

# Production Prometheus setup prometheus: global: scrape_interval: 15s evaluation_interval: 15s external_labels: cluster: us-west-prod region: us-west-2 # Scrape configurations scrape_configs: - job_name: 'kubernetes-pods' kubernetes_sd_configs: - role: pod relabel_configs: - source_labels: [__meta_kubernetes_pod_label_app] target_label: app # Alert rules rule_files: - '/etc/prometheus/rules/*.yml' # Alertmanager alerting: alertmanagers: - static_configs: - targets: ['alertmanager:9093']

Key Principles for Scale

  • Federated Prometheus: One Prometheus per cluster, then federate to central. Prevents single point of failure.
  • Storage strategy: Local storage + long-term remote storage (Thanos, Cortex). Don't rely on single disk.
  • Retention policy: Keep 2-4 weeks locally, 1+ year remotely.
  • Dimensionality control: High cardinality labels create storage explosions. Label thoughtfully.
The Cardinality Trap: Labeling every request ID as a dimension will blow up your storage. Use labels for dimensions (environment, service, type) not for unique values (user ID, request ID).

Logs: ELK Stack at Scale

Logs are important but expensive at scale. Here's a scalable approach:

# Production ELK pipeline Kubernetes Pods ‚Üí Fluent Bit (lightweight agent) ‚Üí Kafka (buffering layer) ‚Üí Logstash (enrichment) ‚Üí Elasticsearch (storage) ‚Üí Kibana (querying)

Cost Optimization Strategies

  • Sampling: Log 100% of errors, 10% of info, 1% of debug
  • Filtering: Don't log health checks, expected errors, or noise
  • Aggregation: Move logs to cheaper tiers after 30 days
  • Index rotation: Create daily indices, archive old ones

Structuring Logs for Queryability

# Good: Structured JSON logs { "timestamp": "2026-02-06T10:30:00Z", "level": "ERROR", "service": "payment-api", "request_id": "abc-123", "user_id": "user-456", "error_type": "PaymentGatewayTimeout", "duration_ms": 5000, "trace_id": "xyz-789" } # Bad: Unstructured text "Failed to process payment for user 456, error: timeout after 5000ms"

Traces: Distributed Tracing with Jaeger

Traces show the complete journey of a request through your system:

Instrumentation Pattern

# Service A calls Service B # Service A (Node.js with Jaeger): const tracer = initTracer({ serviceName: 'service-a', sampler: { type: 'const', param: 1 }, reporter: { logSpans: true } }); const span = tracer.startSpan('process_order'); await callServiceB(span); span.finish(); // Service B automatically receives trace context // Both services contribute to same trace

Sampling Strategy

Tracing everything is expensive. Sample strategically:

  • Always sample: Errors (100%), slow requests (10%), customer interactions (5%)
  • Never sample: Non-critical internal calls, health checks
  • Dynamic sampling: Increase during incidents, decrease during calm

Connecting the Three Pillars

The magic happens when you correlate all three:

  • Metrics alert: "CPU spike detected"
  • Logs explain: "Service X is memory leaking"
  • Traces show: "Request Y takes 3x longer through Service X"

Use trace IDs in logs to connect them. Use request IDs in metrics to drill down.

Alerting: The Critical Piece

Observability without alerting is just dashboarding. You need proactive alerts:

# AlertManager example alert: HighErrorRate expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.05 for: 5m annotations: summary: "High error rate detected" severity: "critical" alert: PodCrashLooping expr: rate(kube_pod_container_status_restarts_total[15m]) > 0.1 for: 5m annotations: summary: "Pod is crash looping" runbook: "https://wiki.internal/runbooks/pod-crash-loop"

The Observability Maturity Journey

Level 1 (Basic): Prometheus + Grafana dashboards. You know your system's state.

Level 2 (Intermediate): Add ELK for logs. You can debug issues.

Level 3 (Advanced): Add Jaeger for traces. You understand latency.

Level 4 (Mature): All three + smart alerting + runbooks. Your team reacts to issues, not searches for them.

Most teams are stuck at Level 1 or 2. Getting to Level 3-4 is the difference between managing infrastructure and running it smoothly.
