The Challenge: From Manual to GitOps

Before GitOps, we had the typical operational problems:

  • Manual deployments: Engineers SSH'ing into servers, running scripts
  • Configuration drift: Cluster state diverged from desired state
  • No audit trail: Who changed what? When? Why? Unknown
  • Disaster recovery nightmare: 4-6 hours to recover from failures
  • Team isolation issues: One team could break another's workloads

We needed a system where:

  • Git is the single source of truth
  • Changes are auditable and reversible
  • Teams are isolated but not siloed
  • Disaster recovery is automatic
  • Deployment is fast and reliable

Architecture: App-of-Apps Pattern

The foundation is Argo CD's "App-of-Apps" pattern‚Äîa single root application that manages other applications:

This pattern solves a critical bootstrapping problem: How does Argo manage itself? By making it self-healing and declarative through Git.

Team Organization: Argo Projects with RBAC

The magic happens at the team level. Each team gets an isolated Argo Project with enforced boundaries:

Project Structure

platform-tools (Admin only) - Namespaces: vault, external-secrets, argocd - Purpose: Core infrastructure - Repos: Approved internal repos only platform-observability (DevOps Team) - Namespaces: observability - Purpose: Monitoring and alerting - Repos: Approved monitoring configs platform-security (Security Team) - Namespaces: security - Purpose: RBAC and policies - Repos: Approved security configs workloads (Development Teams) - Namespaces: workloads, applications - Purpose: User applications - Repos: Any approved application repo

Each project is locked down by:

  • Source repositories: Only approved Git repos can deploy
  • Destination namespaces: Projects can only write to their namespaces
  • Resource types: Projects can only deploy allowed resource types
Key Insight: RBAC in Argo Projects prevents "blast radius." One team's misconfiguration cannot affect another team's workloads because Kubernetes namespace isolation + Argo RBAC enforcement creates hard boundaries.

Secrets Management: Vault + External Secrets Operator

Never store secrets in Git. We use Vault + External Secrets Operator:

Git Repository ‚Üì External Secrets Operator watches for ExternalSecret resources ‚Üì External Secrets queries Vault ‚Üì Vault returns secret ‚Üì ESO creates Kubernetes Secret in target namespace

This approach means:

  • Secrets never exist in Git (audit-friendly)
  • Secrets are injected at deployment time
  • Easy secret rotation without code changes
  • Secrets scoped to teams via Vault policies

Disaster Recovery: 15-Minute Recovery Time Objective

Our disaster recovery is completely automated:

Recovery Process

1. Failure detected (5 minutes via monitoring) 2. Automated backup restore initiated (3 minutes) 3. Argo detects "out of sync" and auto-syncs (5 minutes) 4. Workloads resume (2 minutes) Total: ~15 minutes

This is possible because:

  • All state is in Git (recover by re-running Argo sync)
  • Vault secrets are separate from cluster state
  • Kubernetes objects are declarative and idempotent
  • Automated recovery scripts handle the orchestration

Production Lessons Learned

1. Start Simple, Scale Gradually

Don't deploy everything via GitOps day one. Start with non-critical workloads, prove the system works, then expand.

2. Namespace-First Thinking

Design your RBAC around namespaces, not roles. Each team gets a namespace (or few), everything else follows.

3. Sync Strategy Matters

Manual sync for infrastructure (requires approval). Automatic sync for applications (fast feedback).

4. Observability is Critical

Monitor Argo itself. Track sync success rate, deployment frequency, lead time for changes. These metrics matter for SLOs.

5. Git Workflow Discipline

Enforce code reviews for infrastructure changes. A single bad Terraform file or Kubernetes manifest can break everything.

Metrics: Why This Matters

  • Deployment Frequency: 50+ per week (vs 2-3 before GitOps)
  • Lead Time for Changes: 15 minutes (vs 2-3 days before)
  • Mean Time to Recovery: 15 minutes (vs 4-6 hours before)
  • Change Failure Rate: <5% (vs 15-20% before)
  • Deployment Reliability: 99.99% success (vs 85-90% before)

These improvements compound. Faster deployments with higher reliability enable faster innovation and fewer incidents.

The Path Forward

GitOps is not a tool‚Äîit's a philosophy. Once you embrace "Git as truth," everything else falls into place. Your infrastructure becomes version-controlled, auditable, and reproducible. Recovery is just a git rollback away.

The production platform I built proves this works at real scale. Hundreds of deployments per week, multiple teams, zero manual deployments, 99.99% uptime.
