Resource Management: The Foundation of Scale

Every Kubernetes cluster has finite resources. Requests and limits are your primary tool for managing them effectively.

Understanding Requests vs Limits

Requests are what your pod guarantees to get. Limits are the ceiling it cannot exceed. Setting these correctly is crucial:

apiVersion: v1 kind: Pod metadata: name: web-app spec: containers: - name: app image: myapp:1.0 resources: requests: memory: "256Mi" cpu: "100m" limits: memory: "512Mi" cpu: "500m"

This configuration reserves 256Mi of memory and 100m of CPU. If the pod needs more (up to the limit), it can use it. If it exceeds the limit, Kubernetes terminates it.

Pro Tip: Set requests based on typical usage, limits based on worst-case. Underestimating requests causes scheduling failures; oversizing wastes cluster resources.

Horizontal Pod Autoscaling

Static pod counts are a relic of the past. Use HPA to scale based on metrics:

apiVersion: autoscaling/v2 kind: HorizontalPodAutoscaler metadata: name: web-app-hpa spec: scaleTargetRef: apiVersion: apps/v1 kind: Deployment name: web-app minReplicas: 3 maxReplicas: 50 metrics: - type: Resource resource: name: cpu target: type: Utilization averageUtilization: 70

This scales from 3 to 50 replicas based on CPU utilization. When average CPU hits 70%, new pods are added. When it drops, pods are removed.

Networking: Beyond Defaults

Default Kubernetes networking is permissive—any pod can talk to any other. In production, you need fine-grained control.

Network Policies

Network policies define what traffic is allowed between pods:

apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: deny-all spec: podSelector: {} policyTypes: - Ingress --- apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: allow-frontend-to-api spec: podSelector: matchLabels: app: api policyTypes: - Ingress ingress: - from: - podSelector: matchLabels: app: frontend

Start with "deny all", then explicitly allow necessary communication. This principle of least privilege prevents lateral movement during security incidents.

Security: Beyond Default Service Accounts

Default Kubernetes security is permissive. Production clusters need hardening:

Pod Security Policies

Enforce security standards across workloads. Disable privileged containers, enforce read-only filesystems, require security contexts:

apiVersion: policy/v1beta1 kind: PodSecurityPolicy metadata: name: restricted spec: privileged: false allowPrivilegeEscalation: false requiredDropCapabilities: - ALL volumes: - 'configMap' - 'emptyDir' - 'projected' - 'secret' - 'downwardAPI' - 'persistentVolumeClaim' hostNetwork: false hostIPC: false hostPID: false runAsUser: rule: 'MustRunAsNonRoot' seLinux: rule: 'MustRunAs' supplementalGroups: rule: 'RunAsAny' fsGroup: rule: 'RunAsAny' readOnlyRootFilesystem: true

This policy prevents the most common security misconfigurations. Enforce it via admission controllers.

Observability: Visibility Into Everything

You cannot operate what you cannot observe. Three pillars form the foundation:

Metrics

Deploy Prometheus and Grafana. Key metrics to monitor:

  • Pod metrics: CPU, memory, network I/O
  • Node metrics: Capacity, pressure, disk space
  • Cluster metrics: API server latency, etcd health
  • Application metrics: Request latency, error rate, business metrics

Logs

Send logs to ELK, Loki, or cloud providers. Structure logs as JSON for queryability:

{"timestamp":"2026-02-03T10:30:00Z","level":"ERROR","service":"api","request_id":"abc123","message":"Database connection failed","error":"timeout"}

Traces

Use Jaeger or similar for request tracing. See how requests flow through your microservices and identify bottlenecks.

Cost Optimization at Scale

Kubernetes clusters can consume resources (and money) rapidly without proper controls:

Resource Quotas

Prevent any single team from consuming all cluster resources:

apiVersion: v1 kind: Namespace metadata: name: team-a --- apiVersion: v1 kind: ResourceQuota metadata: name: team-a-quota namespace: team-a spec: hard: requests.cpu: "100" requests.memory: "200Gi" limits.cpu: "200" limits.memory: "400Gi" pods: "100"

Spot Instances & Cost Optimization

Use spot instances for non-critical workloads. They're 70-90% cheaper than on-demand. Use node affinity to ensure critical services run on reliable nodes:

The Path Forward

Operating Kubernetes at scale requires discipline, but the reward is systems that scale effortlessly with demand. Start with the fundamentals: proper resource management, network policies, security policies, and observability. Build from there.

The best Kubernetes clusters are often the least visible—they work so well that teams forget they're running Kubernetes at all.
