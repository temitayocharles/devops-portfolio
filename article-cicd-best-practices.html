CI/CD Best Practices: Building Reliable Deployment Pipelines

Author: Temitayo Charles Published: February 9, 2026 Read Time: 13 minutes

CI/CD is the backbone of modern DevOps. When done right, it enables teams to deploy code hundreds of times per day with confidence. When done wrong, it becomes a nightmare of broken deployments, security vulnerabilities, and firefighting.

I've built CI/CD pipelines that handle 50+ deployments per week with zero manual intervention and 99%+ success rate. This article shares the practices that make it work at production scale.

The Pipeline Architecture

A production CI/CD pipeline has distinct stages, each with specific responsibilities:

Commit ‚Üí Build ‚Üí Unit Test ‚Üí Security Scan ‚Üí Deploy Dev ‚Üì 2m 5m 2m 3m Integration Test ‚Üí Deploy Staging ‚Üí Approval ‚Üí Deploy Prod 5m 5m manual 10m Total Average Pipeline Time: 32 minutes

Key Principles

  • Fast feedback: Unit tests in < 2 minutes
  • Parallel execution: Tests run concurrently, not sequentially
  • Single artifact: Build once, deploy everywhere
  • Progressive deployment: Dev ‚Üí Staging ‚Üí Prod
  • Automatic rollback: Failed health checks trigger rollback

Build Stage: Create Artifacts

Create versioned, immutable artifacts that flow through the pipeline:

# GitHub Actions build job - name: Generate version run: echo "VERSION=${{ github.run_id }}" >> $GITHUB_ENV - name: Build Docker image run: docker build -t myapp:${{ env.VERSION }} . - name: Push to registry run: docker push myapp:${{ env.VERSION }} # Result: Single immutable image for all environments

Test Stage: Multiple Levels

Level 1: Unit Tests (< 2 minutes, run in parallel)

pytest tests/unit/ --cov=src --cov-fail-under=80 -n auto

Level 2: Integration Tests (5-10 minutes)

docker-compose -f tests/docker-compose.yml up -d pytest tests/integration/ --timeout=300 docker-compose down

Level 3: Contract/API Tests (2-3 minutes)

Ensure service compatibility with clients and dependencies.

Security Scanning

Shift security left. Scan at build time, not deployment:

# 1. Dependency scanning npm audit --audit-level=moderate # 2. Static code analysis (SAST) sonarqube scan --project=myapp # 3. Docker image scanning trivy image myapp:latest # Fail pipeline if vulnerabilities found

Deployment Strategy: Blue-Green with Canary

Minimize risk with gradual rollouts:

# 1. Deploy canary: 10% traffic kubectl set image deployment/myapp \ myapp=myapp:new-version \ --record # 2. Monitor metrics for 5 minutes # If error rate > 1%, auto-rollback # 3. If stable, shift 50% traffic # Then 100%

Rollback Mechanics

Automatic rollback on failure:

# Health check fails GET /health ‚Üí 500 Error rate spike detected ‚Üí Automatic rollback to previous version ‚Üí Alert team # Manual rollback kubectl rollout undo deployment/myapp

Approval Gates

For non-production: Automatic approval after tests pass

For production: Manual approval + change window validation

# Production deployment requires: - All tests passing - Security scan clean - Manual approval - Change window open (e.g., 9am-5pm weekdays only) - On-call engineer confirmed

Monitoring & Alerting

Monitor the CI/CD system itself:

  • Pipeline success rate: Alert if < 95%
  • Deployment frequency: Trend over time
  • Lead time for changes: Commit to production
  • Mean time to recovery: Failure detection to fix
  • Change failure rate: Deployments causing incidents

Common Pitfalls

Pitfall 1: Deployments taking too long ‚Üí Parallelize stages

Pitfall 2: Brittle tests ‚Üí Use containers, avoid external dependencies

Pitfall 3: Manual steps ‚Üí Automate everything

Pitfall 4: No rollback strategy ‚Üí Practice it regularly

Pitfall 5: Skipping staging ‚Üí Always deploy to staging first

The Golden Rule: If you wouldn't deploy it 50 times a day, your CI/CD isn't automated enough. Automation removes human error and enables confidence.

Tools & Platforms

  • Deployment: Kubernetes, Argo CD, Helm, Terraform
